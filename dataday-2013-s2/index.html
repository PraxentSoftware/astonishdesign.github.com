The cost of bootstrap

  • Hooks allow modules to customize behavior
  • Drupal calls hooks during each stage of bootstrap
  • More modules, more hooks, slower bootstrap
One of Drupal's primary strengths is also a weaknesses: a dynamic call stack. Drupal relies on hooks to run custom code that can be included in community contributed modules. This is why Drupal is a rapid development platform; It is composed of modular, reusable components. However, this architecture is also inefficient, and tends to generate massive call stacks.

Massive Callstacks

Bootstrap on Every Request

Avoiding the Bootstrap

Let us compound this problem by further stipulating that you will be integrating data from an external source (external API, feeds, etc.). Just to kick it up a notch, you also want to integrate real time data. There is no way you can feed real time data into the bootstrap.

bootstrap once to load the page and deliver a JS app that will request data from a light weight proxy to the de-normalized data store.

API and Node.JS

Instead of asking Drupal to read data from a relational database, use the client web browser, the front end, to talk to a custom API. Ideally, this API is non-blocking, and asynchronous.

You write an API service that can talk to other services, i.e. a proxy. You can build the proxy in just about anything. The point is, it isn't processing data and it should be able to handle lots of connections. That way, your service acts as fast shuttle of data that bypasses Drupal's bootstrap.

CORS

An API service is necessary because of the single origin policy. Requests for data sources have to originate from the same domain unless Cross Origin Resource Sharing (CORS) is implemented. The way to get around this is to write a server side application that fetches data from outside sources and then relays that data back to the requesting client side application.

RESTful

Since you are building the API, you can make it any way you want, but following some good basic REST conventions is a good idea.

It will make it easier to integrate with front end frameworks.

Organize your Code

Why Discourse Uses Ember

On the front-end, you could write everything as AJAX calls and jQuery DOM manipulation, but that will get messy very quickly as your application grows in sophistication.

In the interests of maintainability and sanity, use a framework that organizes your code and abstracts away the DOM. There are many to choose from. A new one probably came out this morning.

Ember

We use Ember because it is straightforward to use, powerful, flexible, and still fairly light weight. BackBone has been around for a while and is quite nice, but we like how Ember handles garbage collection and the other goodies it ships with.

Algebraix

For Algebraix, all the data comes from an external source, so it makes sense to run the application on the client side, as opposed to importing the data into Drupal first. It also makes the control panel feel more responsive. No more click and hope while waiting for the interface to receive data. Now we've created a conversation with the user.
